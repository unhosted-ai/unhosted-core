how it works

deep dive

two pieces, no magic. a daemon per device speaks a small http api on port 7777; the cli (and any openai-compatible client) sends prompts to it. inference itself runs inside the model runtime you chose — unhosted is the orchestration layer, not the neural network.

01

single-host pipeline

v0.0.1 baseline: one daemon, one runtime, on one machine. your prompt hits the daemon over loopback. the daemon forwards it to llama.cpp's llama-server (or ollama, or lm studio — same shape), streams the sse tokens back, and never persists a thing. closing the cli wipes nothing because nothing was written.

02

multi-node request routing

v0.0.2 adds peers. each daemon owns a router with a target list — Local plus every paired peer. for each request the router picks one target by priority, falls back to Local on failure, and stamps an x-unhosted-forwarded header so peers don't bounce it back into a loop. throughput parallelizes across requests; a single request still runs on a single gpu.

03

peer discovery & pairing

on the lan, daemons announce themselves over multicast dns (_unhosted._tcp.local.) with their name and ed25519 pubkey. browsing is zero-config. pairing is explicit: one side issues a 5-minute unhosted://pair?… token; the other accepts it. each side stores the other's pubkey. nothing pairs silently, ever.

off-lan, a tiny coordination relay matches both ends' connect requests and tells each one where the other lives. then they punch a direct udp path with quic — encrypted end-to-end, the relay never sees plaintext. if punching fails (symmetric nats), the relay falls back to forwarding ciphertext blindly so you still get connectivity, just with the relay in the path.

04

the lifecycle of one request

a chat completion takes the same path whether it lands locally or on a peer — that's the point of the openai-compatible api: the client never knows. here's what happens between "user hits enter" and "first token visible":

  1. parse + classify. request lands at the daemon. it checks for the x-unhosted-forwarded header (peer-to-peer hop) and the x-unhosted-auth signature header. loopback callers bypass the bearer check; everything else needs one or the other.
  2. replay defense. if the request carries a peer signature, the replay guard looks for (pubkey, timestamp, sig-prefix) in its short-ttl set. seen recently? reject. otherwise insert and continue.
  3. route. router picks Local or a peer based on priority + recent failure stats. forwarded hops always pin to Local — no second-level fanout, no routing storms.
  4. forward. request gets handed to the runtime: an http POST /v1/chat/completions with stream: true. the runtime starts emitting sse data: {…} frames almost immediately.
  5. stream + tee. the daemon parses sse, extracts choices[].delta.content, and pipes tokens straight to the client's response body. nothing is buffered to disk; nothing is logged.
  6. done. runtime sends data: [DONE], daemon closes the response, replay guard ttl eventually evicts the entry. no state survives the request except the conversation kept by the browser in localstorage.
05

the runtime layer — not part of unhosted

unhosted doesn't do any matrix math. it doesn't touch model weights, doesn't decide which token comes next, doesn't load gguf files. the neural network — the actual ml — runs inside whichever runtime you pick. all three speak the same openai-compatible /v1/... shape, so unhosted treats them interchangeably.

llama.cpp :8080

the default. llama-server binary from llama.cpp. metal on apple silicon, cuda/vulkan on linux, cpu fallback. brew install llama.cpp or build from source.

ollama :11434

preferred by many users for the model-management ergonomics. unhosted's auto-detect finds it without config; ollama serve and you're done.

lm studio :1234

gui-first. nice on windows where building llama.cpp is fiddly. enable "local server" in the app, unhosted picks it up.

auto-detection. at unhosted serve startup the daemon probes all three on their default localhost ports. if nothing answers, it prints os-specific install hints (homebrew on macos, the ollama curl script on linux, lmstudio.ai on windows). run unhosted doctor any time to see the same report.

network & security

read the long version

three layers, each with a different threat model. plain http on loopback, signed-request auth on the lan, end-to-end encrypted quic between paired peers. no traffic touches an unhosted server — there isn't one.

01

discovery — mdns on the lan

nodes announce themselves over multicast dns (_unhosted._tcp.local.). the announcement carries the node's name and ed25519 public key. browsing is zero-config — every machine on the same wifi sees every other one within seconds, no router config, no static ips, no central directory.

02

pairing — explicit, out of band

two daemons become trusted peers only after one issues a 5-minute pairing token and the other accepts it. the offer uri (unhosted://pair?…) is shared out of band — qr code, signal, in person. each side stores the other's pubkey. nothing pairs silently.

03

transport — quic with pinned identity

peer-to-peer traffic runs over quic + tls 1.3. each daemon's tls cert is bound to its ed25519 identity key; a mitm can't substitute a different cert because the receiver checks the cert's pubkey against the one stored at pairing. lan traffic also accepts plain http for v0.0.x legacy callers, but it requires the same signed-request header.

04

nat traversal — relay-coordinated hole punch

two paired peers on different home networks find each other through a tiny coordination relay. the relay matches both sides' connect requests and tells each one where the other is, then they punch a direct udp path with quic. the relay never sees plaintext — payloads are encrypted end-to-end. if punching fails (symmetric nats), the relay falls back to forwarding ciphertext blindly.

05

request auth — signed, replay-proof

every sensitive request from a paired peer carries an x-unhosted-auth header: <pubkey>:<timestamp>:<ed25519-sig>. the receiver verifies the signature, checks the pubkey against its pair registry, and rejects timestamps outside a short window. a per-node replay guard remembers recently-seen signatures so the same one can't be reused.

06

local boundary — bearer token off loopback

when the daemon binds to 127.0.0.1, the cli and the desktop shell on the same machine talk to it freely. when it binds to 0.0.0.0 for phone access, anything off loopback that isn't a signed peer needs a bearer token printed once at startup. no telemetry, no prompt logging — the daemon persists nothing about your conversations.

  • what's shipped: mdns discovery, pairing, signed-request auth, replay guard, bearer-token boundary, quic peer transport with pubkey-bound certs (diagnostic in v0.0.4).
  • what's next: wireguard-style trusted pairing (v0.1.0), hardened nat traversal, formal threat-model review before v0.1.0 ships.
  • what we don't do: store prompts, run analytics, broker keys, operate any server that sees plaintext. disclose a vulnerability.

use it as a backend

compatible clients

the daemon speaks the openai http api at /v1/chat/completions and /v1/models. anything that knows how to talk to openai, ollama, or llama-server can point at unhosted instead. the routing layer + trust radius are invisible from the client's perspective — it sees a fast openai-compatible server, that's the point.

streaming chat completion 
curl -sN http://127.0.0.1:7777/v1/chat/completions \
    -H 'content-type: application/json' \
    -d '{
      "model": "llama3.2:1b",
      "messages": [{"role":"user","content":"hello"}],
      "stream": true
    }'
data: {"choices":[{"delta":{"content":"hi"}}], ...}
data: {"choices":[{"delta":{"content":" there"}}], ...}
data: [DONE]
  • delta browser point its provider settings at http://127.0.0.1:7777. multi-node routing, peers, future public swarm — all transparent.
  • langchain / llamaindex use any "openai-compatible local endpoint" recipe. set OPENAI_API_BASE to the daemon's url.
  • openwebui add unhosted as an openai-style provider. streaming + tool calls forwarded straight through.
  • aider, cursor (local mode), continue anything that auto-discovers ollama on :11434 or llama-server on :8080 drops onto :7777 the same way.

questions

long-form docs

the questions we get most. short answers. links to the longer write-up where one exists.

does unhosted train models?

no. training is meta's / mistral's / qwen's / alibaba's job — far too expensive for us. we run inference on whatever open-weights model you point us at. you can fine-tune locally with axolotl, mlx-finetune, or llama.cpp itself, then load the result.

does the daemon log my prompts?

no. prompts pass through the daemon to the runtime in memory and disappear after inference. nothing is written to disk by unhosted. the conversation you see in the chat ui is kept in your browser's localstorage, on your device, and you can wipe it at any time.

is my data sent to any server you operate?

we do not operate any server that handles your prompts. the only optional server in the architecture is the coordination relay for off-lan peer connections, and it only ever sees ciphertext — payloads are end-to-end encrypted between paired peers.

do i have to use llama.cpp?

no. unhosted speaks the openai-compatible api, so it talks to llama.cpp's llama-server, ollama, or lm studio equally well. on startup the daemon probes all three on their default ports and uses whichever it finds. unhosted doctor tells you what's running.

how is this funded?

self-funded. no vc money, no token sale. the public-mode swarm (v0.3.0+) settles payments in usdc between users and providers; unhosted takes no cut at the protocol level. see the manifesto for the long version.

why agpl-3.0?

it's the license that prevents someone from taking the code, hosting it as a paid saas, and pretending they wrote it. you can read it, fork it, audit it, ship it. you can sell support, custom integrations, or hardware around it. you can't quietly close-source it as a service.

windows support?

cli + desktop shell ship as a windows binary in every release. gpu acceleration on windows is "designed" — we'll harden it after macos and linux are stable. lm studio is the easiest runtime to use on windows today.

raspberry pi?

aarch64 linux binaries ship in every release. raspberry pi 5 with 8 gb runs llama3.2:1b at a few tokens/sec on cpu — fine for ambient/automation use, slow for chat. a pi mesh on the lan distributes parallel requests cleanly.

how do i contribute?

open an issue or pr at github.com/unhosted-ai/unhosted-core. brand voice rules live in brand.md — plain language, no marketing, no superlatives. security reports go through security.md.
