understand unhosted in five minutes

unhosted is open-source software that turns the computers you already own into one AI inference cluster. this page explains what that means in plain words.

on this page what is it the three modes how it works quickstart what's working today faq where to go next

what is it

most ai today runs in someone else's data center. you send a prompt over the internet, you get tokens back, you pay per token, and the company on the other end keeps a copy of what you asked.

unhosted is the opposite shape. it pools the devices already in your home or your team's office — macbook, gaming pc, home server, raspberry pi — into a single inference cluster. you point one cli at the cluster, you ask a model a question, the model runs locally, the answer comes back. no api key. no rate limit. no logging by default.

when your local hardware can't handle a request — a model too big, a prompt too long — unhosted can optionally tap a public swarm of strangers' gpus, paid in stablecoin per token. that's a safety net, not the default. you can use unhosted forever and never spend a dollar.

the three modes

unhosted has three concentric rings of trust. you decide how far the radius goes for any given request.

local — free, private, default

devices you own, on a network you control. nothing leaves your house. no internet required after the model is downloaded. this is the mode the manifesto is about.

trusted — free, encrypted, opt-in

devices owned by people you've explicitly paired with: a roommate, a family member, a teammate. traffic between paired peers is end-to-end encrypted. no public exposure, no payment.

public — pay-per-token, opt-in

strangers' idle gpus rented in exchange for usdc per token. you set a price ceiling. providers stake collateral and lose it if they cheat. this mode only kicks in when you ask for it — never silently.

how it works

two pieces, no magic.

  1. a daemon — the program that runs on each of your devices. it talks to a local inference backend (today: llama.cpp's llama-server) and exposes a small http api on port 7777.
  2. a cli — unhosted, the command you type. it sends prompts to a daemon and prints the streaming reply.

quickstart

v0.0.1 wraps llama.cpp's existing llama-server. you need llama.cpp installed and a model file before unhosted has anything to talk to.

  1. install llama.cpp

    brew install llama.cpp

    on linux, build it from github.com/ggerganov/llama.cpp. windows works too, with extra steps.

  2. get a small model

    mkdir -p ~/.cache/unhosted/models
cd ~/.cache/unhosted/models
curl -fL -o llama-3.2-1b.gguf \
  https://huggingface.co/bartowski/Llama-3.2-1B-Instruct-GGUF/resolve/main/Llama-3.2-1B-Instruct-Q4_K_M.gguf

    that's a 770 mb file. start small. you can swap in a larger model later.

  3. start the inference backend

    llama-server -m ~/.cache/unhosted/models/llama-3.2-1b.gguf \
  --port 8080 -c 2048 -ngl 99

    -ngl 99 offloads all layers to gpu (metal on apple silicon, cuda on nvidia). leave that off if you don't have one.

  4. build unhosted

    git clone https://github.com/unhosted-ai/unhosted-core
cd unhosted-core
cargo build --release

    requires rust stable 1.75+. if you don't have rust: curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh.

  5. run a node

    ./target/release/unhosted serve

    leave this running in one terminal. it'll log when it's ready.

  6. send a prompt

    ./target/release/unhosted run "explain quantum tunneling in two sentences"

    tokens stream to your terminal as they're generated. that's the whole demo.

faq

is this a crypto project?

no. there is no token, no airdrop, no chain we operate. the public mode (when it ships) uses usdc on existing networks because that's how you do permissionless global payments without a us llc and a stripe account — the same reason a freelancer in argentina takes usdc. you can use the local and trusted modes for the rest of your life and never touch a stablecoin.

are my prompts private?

local mode: yes. nothing leaves your hardware.
trusted mode: yes. traffic to paired peers is end-to-end encrypted, and you only pair with people you actually trust.
public mode: no, by default. you're sending the prompt to a stranger's gpu. when this mode ships, the cli will warn you before each public-mode request. don't send your medical history through it.

why agpl-3.0 instead of mit / apache?

agpl prevents a company from taking unhosted, hosting it as a paid service, and not giving improvements back. that's the entire ethos of the project — we don't want a hosted competitor masquerading as unhosted. you can still read it, fork it, audit it, and ship it for any other purpose.

what hardware do i actually need?

at v0.0.1: anything that runs llama.cpp. an m-series macbook. a linux box with or without a gpu. a windows machine with cuda. small models (1–3b parameters) run on a raspberry pi 5. larger models (70b+) need 24+ gb of vram or a recent apple silicon mac with enough unified memory. real benchmarks land in benchmarks/ as we collect them.

how is this different from ollama or exo?

ollama is great for single-machine local inference and we use llama.cpp under the hood the same way ollama does. unhosted's value is in the cluster layer — pooling several devices, trusted-peer pairing over the internet, and the optional public swarm. exo is the closest thing in spirit; we differ in stack (rust vs python), in the explicit trust-radius model, and in the planned payment layer for public mode.

when can i use it?

v0.0.1 ships today: single machine. v0.0.2 ships when two-node lan cluster is solid — weeks, not months. trusted-peer over the internet is v0.1.0. the public swarm is v0.3.0+. no firm dates. we'll ship and tell you what works.

how do i help?

star the repo, run it on your hardware, file issues that include exact commands and exact error output, send pull requests against open issues. see contributing.
