understand unhosted in five minutes

unhosted is open-source software that turns the computers you already own into one AI inference cluster. this page explains what that means in plain words.

what is it

most ai today runs in someone else's data center. you send a prompt over the internet, you get tokens back, you pay per token, and the company on the other end keeps a copy of what you asked.

unhosted is the opposite shape. it pools the devices already in your home or your team's office — macbook, gaming pc, home server, raspberry pi — into a single inference cluster. you point one cli at the cluster, you ask a model a question, the model runs locally, the answer comes back. no api key. no rate limit. no logging by default.

when your local hardware can't handle a request — a model too big, a prompt too long — unhosted can optionally tap a public swarm of strangers' gpus, paid in stablecoin per token. that's a safety net, not the default. you can use unhosted forever and never spend a dollar.

the three modes

unhosted has three concentric rings of trust. you decide how far the radius goes for any given request.

local — free, private, default

devices you own, on a network you control. nothing leaves your house. no internet required after the model is downloaded. this is the mode the manifesto is about.

trusted — free, encrypted, opt-in

devices owned by people you've explicitly paired with: a roommate, a family member, a teammate. traffic between paired peers is end-to-end encrypted. no public exposure, no payment.

public — pay-per-token, opt-in

strangers' idle gpus rented in exchange for usdc per token. you set a price ceiling. providers stake collateral and lose it if they cheat. this mode only kicks in when you ask for it — never silently.

how it works

two pieces, no magic.

  1. a daemon — the program that runs on each of your devices. it talks to a local inference backend (today: llama.cpp's llama-server) and exposes a small http api on port 7777.
  2. a cli — unhosted, the command you type. it sends prompts to a daemon and prints the streaming reply.

future versions add more daemons across more machines, then encrypted pairing between trusted peers, then a discovery layer for the public swarm. the cli ↔ daemon contract stays the same.

what we do, and what we don't

this matters for understanding the trust model and what to expect.

unhosted is the orchestration layer

it doesn't do any matrix math, doesn't touch the weights, doesn't decide which token comes next. the neural network — the actual ml — runs inside llama.cpp on your hardware. unhosted routes requests to it, multiplexes across nodes, streams results back, and handles the trust and payment layers around all of that.

multi-node is request distribution, not tensor parallelism

today, the multi-node layer distributes whole requests across machines. each request still runs on one gpu at a time. the win is parallel throughput across requests, automatic failover, and the trust architecture — not faster single-request inference. true tensor parallelism (splitting a single model across multiple gpus) is v0.0.4+ via llama.cpp's rpc backend.

what unhosted doesn't do at the daemon level

  • train models — too expensive, that's meta / mistral / qwen / alibaba's job. we run inference on whatever open-weights models exist.
  • modify weights — would break the model and require trust we don't have. you can fine-tune locally with other tools (axolotl, mlx-finetune, llama.cpp itself) and then point unhosted at the result.
  • add content filters — alignment is the model's responsibility. each open-weights model ships with its own training-time alignment; we don't bolt extra filters on top. swap to a different model if a given one's behavior doesn't match what you need.
  • log prompts — they go to llama.cpp on your machine and disappear after inference. nothing is persisted by the daemon except the conversation history your browser keeps in localstorage on your device.

your jurisdiction, your responsibility

"no daemon-level filters" is not the same as "no rules apply." legal use is yours.

different countries treat ai output differently — the eu ai act, the uk online safety act, china's generative ai regulations, and various us state laws all impose specific obligations on operators or users. some content (csam, defamation, certain incitement) is illegal everywhere regardless of what produced it. running open-source software doesn't change any of that, same as compiling gcc doesn't grant immunity from what you write in c.

if you operate a public-mode provider node in v0.3.0+, you'll likely need to apply your own content policies and refusal logic appropriate for your jurisdiction. that policy layer is an operator concern, not a daemon concern; the plugin surface mentioned below is where it can live.

the deeper-ml stuff lives in a separate plugin

this division — core stays minimal, ml-shaped features go elsewhere — is deliberate. things like model swapping at runtime, alternative inference backends (vllm, mlx, candle), per-model system-prompt overrides, fine-tune hooks, quantization tools, and verifiable inference proofs will land as a separate plugin the daemon can load. opt-in, separately versioned, not part of the core binary.

name and shape tbd. the spec gets its own adr when we're closer to building it; today the daemon stays small and the seams for plugins are designed to come later cleanly. expect the plugin surface no earlier than v0.2.0.

using unhosted as a backend for other tools

the daemon speaks the openai http api at /v1/chat/completions and /v1/models. anything that knows how to talk to openai / ollama / llama-server can point at unhosted instead.

$ curl -sN http://127.0.0.1:7777/v1/chat/completions \
    -H 'content-type: application/json' \
    -d '{
      "model": "llama3.2:1b",
      "messages": [{"role":"user","content":"hello"}],
      "stream": true
    }'
data: {"choices":[{"delta":{"content":"hi"}}], ...}
data: {"choices":[{"delta":{"content":" there"}}], ...}
...

known clients that work as-is:

  • delta — privacy-first ai browser. point it at http://127.0.0.1:7777 in delta's provider settings; everything below (multi-node routing, trusted peers, future public swarm) becomes transparent infrastructure.
  • any langchain / llamaindex / openwebui setup configured for "openai-compatible local endpoint"
  • any tool that auto-discovers ollama (:11434) or llama-server (:8080) — pointing it at unhosted (:7777) drops in the same way

the routing layer + trust radius are invisible from the client's perspective. it just sees a fast openai-compatible server. that's the point.

quickstart

v0.0.1 wraps llama.cpp's existing llama-server. you need llama.cpp installed and a model file before unhosted has anything to talk to.

  1. install unhosted

    macos / linux:

    curl -fsSL https://raw.githubusercontent.com/unhosted-ai/unhosted-core/main/scripts/install.sh | sh

    windows: open powershell and run

    irm https://raw.githubusercontent.com/unhosted-ai/unhosted-core/main/scripts/install.ps1 | iex

    auto-detects mac arm64, linux x86_64/arm64, and windows x86_64. installs both the unhosted daemon and the unhosted-desktop native shell. on linux it also drops a launcher entry; on windows it adds a start-menu shortcut; on macos it fetches the .app bundle into /Applications. set UNHOSTED_INSTALL_DIR to install elsewhere, or UNHOSTED_NO_DESKTOP=1 to skip the gui. or build from source: cargo build --release.

    linux runtime deps for the gui (tao+wry uses webkit2gtk under the hood):
    debian/ubuntu: sudo apt install libgtk-3-0 libsoup-3.0-0 libwebkit2gtk-4.1-0
    fedora: sudo dnf install gtk3 libsoup3 webkit2gtk4.1
    arch: sudo pacman -S gtk3 libsoup3 webkit2gtk-4.1

  2. install llama.cpp

    brew install llama.cpp

    on linux, build it from github.com/ggerganov/llama.cpp. windows works too, with extra steps.

  3. pull a model

    unhosted pull llama3.2:1b

    downloads a known good gguf into ~/.cache/unhosted/models/. unhosted models lists everything available — llama3.2:1b, llama3.2:3b, llama3.1:8b, qwen2.5:0.5b, qwen2.5-coder:7b. you can also pass a direct https url to a gguf.

  4. start the inference backend

    llama-server -m ~/.cache/unhosted/models/Llama-3.2-1B-Instruct-Q4_K_M.gguf \
  --port 8080 -c 2048 -ngl 99

    -ngl 99 offloads all layers to gpu (metal on apple silicon, cuda on nvidia). leave that off if you don't have one.

  5. run the daemon

    unhosted serve

    leave this running in one terminal. log says when it's ready. other unhosted nodes on the same lan will discover this one via mdns automatically — pair them with one click in the app sidebar.

  6. open the app

    open http://127.0.0.1:7777

    chat ui in your browser, or run the native unhosted-desktop app for a window in your dock.

also from your phone (same wifi)

the web ui is mobile-responsive. to reach it from a phone on the same network, restart the daemon bound to 0.0.0.0 instead of loopback:

unhosted serve --addr 0.0.0.0:7777

then on your phone, open http://<your-computer-lan-ip>:7777/ — find the ip with ifconfig (mac/linux) or ipconfig (windows). same chat, same model, same conversation history (per-device localstorage). signed-request auth means only paired devices can use peer routing, but the web ui itself is unauthenticated on your lan — keep this off if your network is shared with people you don't trust.

linux: run it as a background service (optional)

drop in a systemd user-service so unhosted starts at login and restarts on crash:

mkdir -p ~/.config/systemd/user
curl -fsSL https://raw.githubusercontent.com/unhosted-ai/unhosted-core/main/scripts/unhosted.service \
  -o ~/.config/systemd/user/unhosted.service
systemctl --user daemon-reload
systemctl --user enable --now unhosted
journalctl --user -u unhosted -f       # follow logs

faq

is this a crypto project?

no. there is no token, no airdrop, no chain we operate. the public mode (when it ships) uses usdc on existing networks because that's how you do permissionless global payments without a us llc and a stripe account — the same reason a freelancer in argentina takes usdc. you can use the local and trusted modes for the rest of your life and never touch a stablecoin.

are my prompts private?

local mode: yes. nothing leaves your hardware.
trusted mode: yes. traffic to paired peers is end-to-end encrypted, and you only pair with people you actually trust.
public mode: no, by default. you're sending the prompt to a stranger's gpu. when this mode ships, the cli will warn you before each public-mode request. don't send your medical history through it.

why agpl-3.0 instead of mit / apache?

agpl prevents a company from taking unhosted, hosting it as a paid service, and not giving improvements back. that's the entire ethos of the project — we don't want a hosted competitor masquerading as unhosted. you can still read it, fork it, audit it, and ship it for any other purpose.

what hardware do i actually need?

at v0.0.1: anything that runs llama.cpp. an m-series macbook. a linux box with or without a gpu. a windows machine with cuda. small models (1–3b parameters) run on a raspberry pi 5. larger models (70b+) need 24+ gb of vram or a recent apple silicon mac with enough unified memory. real benchmarks land in benchmarks/ as we collect them.

how is this different from ollama or exo?

ollama is great for single-machine local inference and we use llama.cpp under the hood the same way ollama does. unhosted's value is in the cluster layer — pooling several devices, trusted-peer pairing over the internet, and the optional public swarm. exo is the closest thing in spirit; we differ in stack (rust vs python), in the explicit trust-radius model, and in the planned payment layer for public mode.

when can i use it?

v0.0.1 ships today: single machine. v0.0.2 ships when two-node lan cluster is solid — weeks, not months. trusted-peer over the internet is v0.1.0. the public swarm is v0.3.0+. no firm dates. we'll ship and tell you what works.

how do i help?

star the repo, run it on your hardware, file issues that include exact commands and exact error output, send pull requests against open issues. see contributing.
